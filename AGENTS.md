# Developer Guide

## Building

```bash
make
```

This will:

1. Generate the tree-sitter parser from `tree-sitter-nasin/grammar.js`
2. Build the nasin binary with `LIB_DIR` set correctly at compile time
3. Copy the binary to `bin/nasin`

## Running

```bash
./bin/nasin build -o <exe> <file>
```

Or to run it directly:

```bash
./bin/nasin run <file>
```

The compiled binary generated by `run` will be placed at `<file dir>/build/<file name>`.

## Snapshot Tests

```bash
make test
```

This runs the snapshot tests using [rere.py](https://github.com/tsoding/rere.py).

### How rere works

rere is a simple snapshot testing tool. It:

1. Runs each command in `tests/_test.list`
2. Captures stdout/stderr
3. Compares against stored snapshots in `tests/_test.list.bi`

To update snapshots after changes:

```bash
make record-test
```

## Project Structure

- `src/` - Compiler source code (Rust)
- `library/` - Standard library (`core.nsn`)
- `tests/` - Test files (`.nsn`) and expected outputs
- `bin/nasin` - Compiled binary
- `cranelift-shim/` - Workspace member wrapping cranelift for developer convenience
- `tree-sitter-nasin/` - Workspace member with tree-sitter grammar

## Key Commands

| Command            | Description               |
| ------------------ | ------------------------- |
| `make`             | Build the project         |
| `make test`        | Run all tests             |
| `make record-test` | Record new test snapshots |
| `make clean`       | Clean build artifacts     |

## Debugging

The project uses gimli for DWARF debug info. To verify debug info:

```bash
readelf --debug-dump=info <binary>
gdb -batch -ex "info functions" <binary>
```

## Formatting

- Keep the rust code formatted with `cargo fmt` and the JS code with `prettier`.
- Avoid adding small comments to explain what the code does. Code should be
  self-explanatory. Keep comments restricted to function/method documentation and complex
  or non-obvious code.
